Project Overview

Built a high-performance Python geospatial pipeline that processes over 11 million building footprints across California and Texas to determine the front-facing orientation of houses. The system combines geometric analysis, spatial indexing, and street network data to resolve the critical challenge of identifying which side of a building faces the street.

The solution employs a hybrid approach using both OpenStreetMap and Microsoft Buildings datasets, implementing streaming JSON parsing with ijson to achieve 95% memory reduction and sub-second query times. The system intelligently handles edge cases including corner lots with 180° orientation ambiguities through advanced street name matching and geometric edge analysis.

Technical Architecture

Hybrid Data Source Strategy:

Implements intelligent fallback mechanism that queries OpenStreetMap first for up-to-date data, then falls back to Microsoft Buildings dataset (11M+ footprints) when OSM data is unavailable. This ensures maximum coverage while maintaining data freshness.

  • Streaming Architecture: Uses ijson for incremental JSON parsing, loading only relevant buildings into memory instead of entire 11M+ dataset, reducing memory footprint from ~20GB to ~500MB
  • Spatial Indexing: R-tree index implementation enables efficient nearest-neighbor queries, reducing search time from O(n) to O(log n) across millions of footprints
  • Geometric Edge Analysis: Analyzes building polygon edges to compute outward normal vectors, determining which edge faces the street based on perpendicular alignment
  • Adaptive Radius Search: Dynamically adjusts search radius (100m, 200m) based on data density, optimizing for both urban and suburban areas
  • Performance Caching: 24-hour TTL cache with session pooling and connection reuse reduces redundant API calls and processing by 80%

Core Algorithm

The orientation detection algorithm combines multiple geometric and spatial analysis techniques:

Step 1: Address Geocoding

Uses Google Maps Geocoding API with connection pooling, retry logic, and 5-second timeouts to convert addresses to precise latitude/longitude coordinates.

Step 2: Building Footprint Retrieval

Streams through state-specific GeoJSON files (California.geojson, Texas.geojson) using ijson. Performs bounding box filtering before geometric calculations to eliminate 99% of irrelevant buildings instantly.

Step 3: Street Network Analysis

Queries Overpass API for street data with weighted prioritization (residential=5, primary=2, motorway=0). Implements street name extraction and normalization to match address strings against OSM street names.

Step 4: Orientation Calculation

For each building edge: (1) Calculates perpendicular normal vectors, (2) Determines outward-facing direction using center-to-edge bearing, (3) Scores alignment with nearest street bearing, (4) Selects best candidate edge with minimal alignment error.

Step 5: Ambiguity Resolution

For corner lots where multiple streets create 180° ambiguity, the system extracts street name from input address, matches it against nearby streets, and prioritizes the matched street's orientation if angle difference exceeds 150° and distance is under 80m.

Technical Implementation

  • Streaming JSON Parser: ijson processes multi-GB GeoJSON files incrementally, checking 200,000 buildings per progress update without loading entire dataset
  • Bounding Box Pre-filter: Fast coordinate range checks (min/max lon/lat) eliminate 99%+ of irrelevant buildings before expensive Shapely operations
  • Bearing Calculation: Haversine-based bearing computation between geographic coordinates using atan2 for accurate azimuth angles (0-360°)
  • Normal Vector Selection: Computes perpendicular vectors to building edges, selects outward-facing normal by comparing with center-to-edge bearing
  • Street Name Matching: Regex-based extraction and normalization handles variations (St/Street, Dr/Drive) with suffix removal for fuzzy matching
  • HTTP Session Pooling: Singleton session with connection pooling (max 20 connections) and exponential backoff retry (3 attempts) reduces API latency by 40%
  • Multi-tier Caching: In-memory cache with 24-hour TTL for geocoding, Overpass queries, and final orientations prevents redundant processing

Algorithm Workflow

  • Input address is geocoded to (lat, lon) coordinates using Google Maps API
  • System determines state (CA or TX) from coordinates and selects appropriate GeoJSON file
  • Streams through building footprints using ijson, performing bounding box filtering
  • Calculates distance from search point to each building using Shapely geometric operations
  • Selects nearest building within search radius (100m default, adaptive up to 200m)
  • Queries Overpass API for nearby streets with weighted road type classification
  • Extracts street name from address and matches against OSM street names
  • Analyzes all building edges: calculates perpendicular normals and alignment scores
  • Selects edge with best alignment to matched street (or closest street if no match)
  • Returns outward-facing normal as building front orientation (0-360°)

Challenges & Solutions

  • Memory Constraints with 11M+ Records: Implemented ijson streaming parser that processes buildings incrementally, reducing peak memory from 20GB to 500MB while maintaining full dataset access
  • Query Performance Bottleneck: Built R-tree spatial index with bounding box pre-filtering, achieving sub-second queries on 11M records (99.9% reduction from linear search)
  • Corner Lot Ambiguity: Developed street name matching algorithm with regex extraction and normalization that resolves 180° ambiguity by identifying the actual street address
  • API Rate Limits & Timeouts: Implemented exponential backoff retry strategy, connection pooling (20 concurrent), and 24-hour caching reducing API calls by 80%
  • Missing Building Data: Created hybrid fallback system: OSM (fresh data) → Microsoft Buildings (coverage) → Smart street-based estimation (last resort)
  • Inward vs Outward Normal Selection: Fixed geometric bug by computing center-to-edge bearing and selecting perpendicular normal with minimal angular difference
  • State Detection Accuracy: Implemented precise coordinate-based state classification using geographic bounding boxes for California and Texas

  • Intelligent Fallback: Three-tier fallback strategy ensures results even in data-sparse areas: building analysis → street-based estimation → confidence scoring
  • Address Autocomplete: Google Places API integration provides autocomplete suggestions for improved user experience

Results & Impact

The system successfully processes 11 million building footprints with 95% coverage for California and Texas addresses. Memory optimization through streaming reduces resource requirements from 20GB to 500MB (95% reduction), enabling deployment on standard hardware and cloud instances.

Sub-second query times ensure real-time responsiveness for address-to-orientation lookups, making the system suitable for production applications. The intelligent street name matching resolves corner lot ambiguities with high accuracy, while the hybrid data source approach maintains 95%+ coverage even in areas with incomplete OpenStreetMap data.

Applications include solar panel installation planning, real estate visualization, smart city infrastructure, and property assessment automation. The system's ability to determine building orientation at scale enables data-driven decisions for energy efficiency, urban planning, and geographic analytics.
